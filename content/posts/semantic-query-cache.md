Every AI agent that touches the outside world – searching the web, querying a database – pays the same tax on every run: the cost of re-fetching information it has already seen. The standard answer is a cache. The standard cache is an exact-match lookup on the query string. And for AI-generated queries, exact-match caching is almost entirely useless.

This post proposes a unified semantic cache architecture backed by Qdrant that handles both web search queries and SQL queries under the same model, with different invalidation strategies appropriate to each. I will focus on the implementation rather than the survey of caching literature. The post assumes familiarity with vector databases and basic SQL concepts.

## Why Exact-Match Caching Fails

<#why-exact-match-caching-fails>

Consider two queries generated by a research agent thirty minutes apart:

```
Query 1: "transformer attention mechanism explained"
Query 2: "how does self-attention work in transformers"
```

An exact-match cache (Redis, Memcached) sees two distinct keys and makes two API calls. The results are nearly identical. The second call was wasted.

The same problem appears in SQL. A business intelligence tool generates these two queries against the same table on the same day:

```sql
-- Query 1
SELECT region, SUM(revenue) FROM orders
WHERE quarter = 'Q3-2025' GROUP BY region;

-- Query 2
SELECT territory AS region, SUM(sales_amount) AS revenue
FROM transactions WHERE fiscal_quarter = 'Q3-2025'
GROUP BY territory;
```

If `orders` and `transactions` are views over the same underlying fact table, these queries return the same data. An exact-match cache misses entirely. A semantic cache would recognize the near-identical intent and return the first result for the second query.

**GPTCache** ([Bang et al. 2023](https://arxiv.org/abs/2306.02779)) introduced the semantic caching concept for LLM completions: embed the user prompt, search for similar cached prompts, return the cached completion if similarity exceeds a threshold. The key insight is that the cache key is the *embedding* of the query, not the query string itself.

GPTCache targets LLM completions – single-turn, stateless, with no notion of freshness. Query caching for agents is harder: results expire, data sources change, and a stale cache hit is silently wrong. I want to extend the semantic caching idea to handle these properties.

## The Problem is Different for Web Search vs. SQL

<#the-problem-is-different-for-web-search-vs-sql>

Before designing the architecture, it is worth being precise about how the two workloads differ, because the differences drive the most important design decisions.

**Web search** results have *temporal staleness* but *semantic forgiveness*. A cached result for “latest LLM benchmarks” from yesterday is probably still useful today. A result from six months ago is not. The staleness threshold is a function of topic category: breaking news expires in hours, research papers expire in weeks, reference documentation expires in months. Semantic forgiveness means that a cache hit with cosine similarity 0.91 is usually fine – two differently-worded questions about the same topic will accept approximately the same set of search results.

**SQL** results have *data staleness* and *semantic strictness*. A cached result becomes invalid not when time passes but when the underlying table changes. Two queries with cosine similarity 0.91 may return numerically different answers if they join different tables or apply different filters. A false positive cache hit – returning a cached result for a query that is only *similar* but not *equivalent* – produces silently wrong data. This is worse than a cache miss.

These two properties drive the main architectural differences:

|Property            |Web search              |SQL                    |
|--------------------|------------------------|-----------------------|
|Cache hit threshold |~0.92-0.94              |~0.97-0.99             |
|Invalidation trigger|Time (TTL per topic)    |Data change (per table)|
|False positive cost |Minor relevance loss    |Silent wrong answer    |
|Result storage      |Compressed result list  |Pointer to result store|
|Frequency signal    |Useful for TTL extension|Useful for pre-warming |

Both workloads share the same underlying Qdrant infrastructure, but the routing logic and invalidation strategies diverge.

## The Unified Qdrant Schema

<#the-unified-qdrant-schema>

The cache lives in a single Qdrant collection. Each cached query is one point carrying three named vector fields and a structured payload.

### Named Vectors

```python
from qdrant_client import QdrantClient
from qdrant_client.models import VectorParams, Distance

client = QdrantClient(url="http://localhost:6333")

client.create_collection(
    collection_name="query_cache",
    vectors_config={
        "query_nl":      VectorParams(size=1536, distance=Distance.COSINE),
        "query_sql":     VectorParams(size=1536, distance=Distance.COSINE),
        "result_summary": VectorParams(size=1536, distance=Distance.COSINE),
    }
)
```

- `query_nl` – embedding of the natural language question as typed or generated by the agent
- `query_sql` – embedding of the normalized SQL string (web search entries leave this zero-padded). Normalization: lowercase, collapse whitespace, replace literals with `?`, strip comments
- `result_summary` – embedding of a short LLM-generated prose summary of the result set. This enables a third query mode: “find me cached results similar to *this output*,” useful for result deduplication across sessions

### Payload Schema

```python
from qdrant_client.models import PointStruct

point = PointStruct(
    id=cache_entry_id,
    vector={
        "query_nl":       embed(natural_language_query),
        "query_sql":      embed(normalized_sql) if is_sql else zero_vector,
        "result_summary": embed(result_summary_text),
    },
    payload={
        "query_type":          "sql",        # "sql" | "web"
        "source":              "postgres",   # "postgres"|"bigquery"|"serpapi"|"tavily"
        "query_nl_text":       "total revenue by region Q3 2025",
        "query_sql_text":      "select region, sum(revenue) from orders where ?",
        "table_dependencies":  ["orders", "regions"],
        "topic_category":      None,         # web only: "breaking_news"|"research"|"reference"
        "fetched_at":          1716000000,
        "ttl_seconds":         None,         # web only
        "hit_count":           0,
        "last_accessed":       1716000000,
        "result_pointer":      "s3://cache/abc123.parquet",
        "result_row_count":    142,
        "stale":               False,
    }
)
```

The `result_pointer` field is important: actual query results are not stored in Qdrant. Result sets can be millions of rows. Qdrant stores only the metadata and embeddings; the result lives in S3, Redis, or a local disk cache. The pointer is what the cache returns on a hit.

## Web Search Caching: Topic-Aware TTL

<#web-search-caching-topic-aware-ttl>

### TTL Tiers by Topic Category

The single most useful improvement over naive TTL caching is making expiry topic-sensitive. A query classified as `breaking_news` should expire in one hour; a query about `mathematical_foundations` should be valid for thirty days. I use four categories with empirically reasonable defaults:

|Category       |Examples                                               |Default TTL|
|---------------|-------------------------------------------------------|-----------|
|`breaking_news`|“latest GPT-5 release”, “today’s AI funding”           |1 hour     |
|`research`     |“attention mechanism transformers”, “LoRA fine-tuning” |72 hours   |
|`reference`    |“Python list comprehension syntax”, “HTTP status codes”|168 hours  |
|`evergreen`    |“backpropagation derivation”, “Bayes theorem”          |720 hours  |

Classification runs inline before the cache lookup, using a small fast model:

```python
CLASSIFY_PROMPT = """Classify this search query into one of four categories:
- breaking_news: about current events, recent releases, today's news
- research: about ML/AI techniques, papers, methods
- reference: about syntax, APIs, documentation, how-to
- evergreen: about mathematical concepts, foundational theory

Query: {query}
Return only: breaking_news | research | reference | evergreen"""
```

The TTL tiers are stored in a config dict and written into the payload at cache-write time. At cache-read time, the Qdrant query includes a must-filter on freshness:

```python
from qdrant_client.models import Filter, FieldCondition, Range
import time

def web_search_cache_lookup(query: str, threshold: float = 0.93) -> list | None:
    query_embedding = embed(query)
    now = int(time.time())

    results = client.search(
        collection_name="query_cache",
        query_vector=("query_nl", query_embedding),
        query_filter=Filter(must=[
            FieldCondition(key="query_type", match={"value": "web"}),
            FieldCondition(key="stale",      match={"value": False}),
            FieldCondition(
                key="fetched_at",
                range=Range(gte=now - MAX_TTL_ANY_CATEGORY)
            ),
        ]),
        limit=1,
        score_threshold=threshold,
        with_payload=True,
    )

    if not results:
        return None

    hit = results[0]
    ttl = TTL_BY_CATEGORY[hit.payload["topic_category"]]
    if hit.payload["fetched_at"] + ttl < now:
        # Expired under its specific TTL even though within the global window
        client.set_payload(
            collection_name="query_cache",
            payload={"stale": True},
            points=[hit.id],
        )
        return None

    # Update hit metadata
    client.set_payload(
        collection_name="query_cache",
        payload={"hit_count": hit.payload["hit_count"] + 1,
                 "last_accessed": now},
        points=[hit.id],
    )
    return load_result(hit.payload["result_pointer"])
```

The two-pass expiry check is intentional: the Qdrant filter uses `MAX_TTL_ANY_CATEGORY` to pre-filter clearly expired points, and the Python layer applies the per-category TTL. This avoids scanning the full collection while keeping category-specific logic in application code where it is easy to modify.

### Hit Rate and the Threshold Tradeoff

The cache hit threshold is the most consequential parameter. Too low and the cache returns irrelevant results; too high and the cache is no better than exact-match. For web search, I find 0.92-0.94 works well on research agent query distributions. The right way to calibrate is to log query pairs with human-rated relevance equivalence, plot precision-recall at multiple thresholds, and pick the threshold at the desired precision point.

A useful heuristic: if two queries would produce the same top-3 search results, they should be a cache hit. If they would produce different result sets, they should not. Sampling 200 query pairs from real agent logs and rating them takes an afternoon and produces a reliable threshold.

## SQL Caching: Table-Dependency Invalidation

<#sql-caching-table-dependency-invalidation>

SQL caching has a harder invalidation problem. The data in the database changes, and any cached query that touches the changed table may now return wrong results. Most SQL caches handle this by flushing the entire cache on any write – which defeats the purpose for high-write workloads.

The Qdrant architecture enables *surgical invalidation*: when table `orders` is updated, only cache entries with `"orders"` in `table_dependencies` are invalidated.

### Query Normalization and SQL Embedding

Before embedding a SQL query, it is normalized to collapse semantically equivalent forms:

```python
import sqlparse
import re

def normalize_sql(query: str) -> str:
    # Parse and reformat
    parsed = sqlparse.format(
        query,
        reindent=False,
        keyword_case="lower",
        strip_comments=True,
    )
    # Replace literals with placeholders
    parsed = re.sub(r"'[^']*'", "?", parsed)
    parsed = re.sub(r"\b\d+\b", "?", parsed)
    # Collapse whitespace
    return " ".join(parsed.split())
```

The normalized SQL is then embedded using a code-aware embedding model. Standard text embeddings (OpenAI `text-embedding-3-small`) work acceptably for SQL, but models fine-tuned on code produce meaningfully better results – `nomic-ai/nomic-embed-code` or Microsoft’s `codebert-base` give ~4-6% better recall at equivalent precision thresholds on SQL workloads.

### Extracting Table Dependencies

Table dependencies are extracted from the query at cache-write time using `sqlparse` or a proper SQL parser:

```python
import sqlparse
from sqlparse.sql import Identifier, IdentifierList
from sqlparse.tokens import Keyword, DML

def extract_tables(query: str) -> list[str]:
    tables = []
    parsed = sqlparse.parse(query)[0]
    from_seen = False
    for token in parsed.tokens:
        if token.ttype is DML and token.value.lower() == "select":
            continue
        if token.ttype is Keyword and token.value.lower() in ("from", "join"):
            from_seen = True
            continue
        if from_seen:
            if isinstance(token, Identifier):
                tables.append(token.get_real_name())
            elif isinstance(token, IdentifierList):
                for identifier in token.get_identifiers():
                    if isinstance(identifier, Identifier):
                        tables.append(identifier.get_real_name())
            from_seen = False
    return [t for t in tables if t]
```

For complex queries with CTEs, subqueries, and views, a full SQL parser (like `pglast` for PostgreSQL or `sqlglot`) is more reliable than `sqlparse`.

### CDC-Based Invalidation

When a table is updated, a change-data-capture event fires and the invalidation job runs:

```python
def invalidate_on_table_change(table_name: str):
    """Called by CDC consumer when table_name receives a write."""

    # Find all cache entries that depend on this table
    hits, _ = client.scroll(
        collection_name="query_cache",
        scroll_filter=Filter(must=[
            FieldCondition(key="query_type", match={"value": "sql"}),
            FieldCondition(key="stale",      match={"value": False}),
            FieldCondition(
                key="table_dependencies",
                match={"value": table_name}
            ),
        ]),
        limit=1000,
        with_payload=False,
    )

    if hits:
        point_ids = [hit.id for hit in hits]
        client.set_payload(
            collection_name="query_cache",
            payload={"stale": True},
            points=point_ids,
        )
```

This marks entries stale rather than deleting them immediately. Stale entries are excluded from cache lookups by the `stale: False` filter, but retained in the collection for analytics – tracking which queries are frequently invalidated surfaces the tables with the highest write contention, which is useful for database optimization.

A background job runs nightly to hard-delete points that have been stale for more than seven days:

```python
def purge_stale_entries(max_age_seconds: int = 604800):
    cutoff = int(time.time()) - max_age_seconds
    client.delete(
        collection_name="query_cache",
        points_selector=Filter(must=[
            FieldCondition(key="stale", match={"value": True}),
            FieldCondition(key="fetched_at", range=Range(lte=cutoff)),
        ]),
    )
```

### The SQL Threshold Problem

The threshold for SQL cache hits requires more care than for web search. I model the cache lookup as a precision-critical classification: a hit should only fire when the two queries are *semantically equivalent* – they would return the same result set against the same database state.

Empirically, the cosine similarity distribution for SQL queries has a clear bimodal structure: queries that are genuinely equivalent cluster around 0.97-0.99 (they differ only in formatting, alias names, or literal values); queries that are similar but not equivalent (same domain, different filters) cluster around 0.85-0.92. There is a useful gap between 0.92 and 0.97 that makes threshold selection tractable. I use 0.97 as a default and treat any hit below 0.99 as requiring a secondary verification step: run a lightweight query plan comparison to confirm the normalized execution paths match before returning the cached result.

### Frequency-Based Pre-Warming

The `hit_count` and `last_accessed` payload fields enable a pre-warming strategy for high-frequency queries. A nightly job identifies cache entries with `hit_count > N` and `last_accessed` within the past 24 hours, then re-executes them against the live database to refresh the cached result before it is next requested:

```python
def prewarm_hot_queries(min_hit_count: int = 50):
    hot_entries, _ = client.scroll(
        collection_name="query_cache",
        scroll_filter=Filter(must=[
            FieldCondition(key="query_type",  match={"value": "sql"}),
            FieldCondition(key="stale",       match={"value": False}),
            FieldCondition(
                key="hit_count",
                range=Range(gte=min_hit_count)
            ),
        ]),
        with_payload=True,
        limit=500,
    )

    for entry in hot_entries:
        fresh_result = execute_query(entry.payload["query_sql_text"])
        update_result_pointer(entry.id, fresh_result)
        client.set_payload(
            collection_name="query_cache",
            payload={"fetched_at": int(time.time()), "stale": False},
            points=[entry.id],
        )
```

This pattern is most valuable for BI dashboards: the same ten queries run by hundreds of users every morning. Pre-warming at 3am means every user hits a warm cache rather than a cold database.

## The Cache Lookup Pipeline

<#the-cache-lookup-pipeline>

At inference time, the cache intercepts every query before it reaches the database or search API. The pipeline:

```
Incoming query (NL or SQL)
        |
        v
[1] Classify: web or sql?
        |
        +-- web --> [2a] Classify topic category
        |                    |
        |                    v
        |           [3a] Qdrant search on query_nl vector
        |               with freshness filter + TTL check
        |                    |
        +-- sql --> [2b] Normalize SQL + extract tables
                             |
                             v
                    [3b] Qdrant search on query_nl vector
                        AND query_sql vector (RRF fusion)
                        with stale:False filter
                             |
                             v
                    [4b] Threshold check (0.97+)
                        Optional: execution plan verification
        |
        v
HIT: load result from result_pointer
MISS: execute query, write new cache entry, return result
```

The SQL path uses RRF fusion over both `query_nl` and `query_sql` named vectors. Natural language embedding catches intent; SQL embedding catches structural equivalence. A query that is high-similarity on both dimensions is a strong cache hit candidate; a query that is high-similarity on NL but low-similarity on SQL is likely a paraphrase that would produce different results – treat as a miss.

```python
from qdrant_client.models import Prefetch, FusionQuery, Fusion

def sql_cache_lookup(nl_query: str, sql_query: str) -> any:
    nl_emb  = embed(nl_query)
    sql_emb = embed(normalize_sql(sql_query))

    results = client.query_points(
        collection_name="query_cache",
        prefetch=[
            Prefetch(
                query=NamedVector(name="query_nl",  vector=nl_emb),
                limit=20,
            ),
            Prefetch(
                query=NamedVector(name="query_sql", vector=sql_emb),
                limit=20,
            ),
        ],
        query=FusionQuery(fusion=Fusion.RRF),
        query_filter=Filter(must=[
            FieldCondition(key="query_type", match={"value": "sql"}),
            FieldCondition(key="stale",      match={"value": False}),
        ]),
        limit=1,
        with_payload=True,
    )

    if not results or results[0].score < SQL_THRESHOLD:
        return None

    return load_result(results[0].payload["result_pointer"])
```

## Measuring Cache Effectiveness

<#measuring-cache-effectiveness>

Three metrics matter for this cache:

**Hit rate** – the fraction of queries served from cache. For a research agent making 30-50 web searches per session on overlapping topics, I observe 35-45% hit rate after the first few sessions of warmup. SQL hit rate depends heavily on workload: dashboards with fixed query patterns see 70-80%; ad-hoc analytics see 10-20%.

**Precision** – the fraction of cache hits that return a result the user would have accepted if fetched fresh. For web search at threshold 0.93, precision is approximately 0.91 on research-domain queries. For SQL at threshold 0.97, precision exceeds 0.98 on normalized, well-structured queries. Precision degrades for queries involving complex CTEs or dynamic date ranges.

**Latency savings** – a Qdrant lookup takes 5-15ms. A web search API call takes 300-800ms. A cold SQL query against a large table takes 500ms to 30s. The cache is economically justified even at 10% hit rate for slow SQL queries.

$$\text{Effective latency} = (1 - h) \cdot t_{\text{source}} + h \cdot t_{\text{cache}}$$

where $h$ is the hit rate, $t_{\text{source}}$ is the average source latency, and $t_{\text{cache}}$ is the Qdrant lookup latency. At $h = 0.35$, $t_{\text{source}} = 500\text{ms}$, $t_{\text{cache}} = 10\text{ms}$, effective latency drops from 500ms to 328ms – a 34% reduction even at modest hit rates.

## Challenges and Open Problems

<#challenges-and-open-problems>

**Threshold calibration is workload-specific.** The 0.93 and 0.97 thresholds I cite are derived from research agent and BI dashboard workloads. A customer support agent with highly specific product queries will need a different calibration. There is no universal threshold, and the cost of miscalibration is asymmetric: too low produces wrong answers (SQL) or irrelevant results (web); too high produces a cache that never hits. Domain-specific calibration on logged query pairs is unavoidable.

**The SQL normalization problem is not fully solved.** Queries that use different views, aliases, or CTEs to express the same computation will normalize to different strings and produce embeddings that are similar but not identical. A full semantic equivalence check requires a query planner, not just a string normalizer. For production systems serving financial or medical data, the execution-plan verification step is not optional.

**Cache poisoning.** A malicious or buggy agent can write incorrect results into the cache, which are then served to subsequent queries. Web search caching is relatively low-risk; SQL caching of financial data is not. Mitigations: require a quorum of N matching results before writing (too expensive for most workloads), implement a result validation step before writing, or restrict write access to trusted query paths only.

**The freshness-hit-rate tradeoff has no clean solution.** Shorter TTLs improve freshness but reduce hit rate. Longer TTLs improve hit rate but surface stale results. Topic-aware TTL is a heuristic that works well on average but breaks for individual queries that don’t fit neatly into a category. A learned TTL – using historical invalidation rates per query cluster to set per-cluster TTLs – would be more precise but requires significant instrumentation.

**Multi-tenant isolation.** When the cache serves multiple users or projects, their query histories must not leak across tenants. Adding `tenant_id` to every payload filter is straightforward. More subtle: the `result_pointer` must point to tenant-scoped storage. A misconfigured result pointer that serves one tenant’s financial data to another is a serious security issue. The cache infrastructure needs explicit tenant isolation testing.
